# Interview Practice Maintenance Rules

## How to Write Notes

1. **One interview concept per file** - keep problems, designs, and behavioral patterns focused.
2. **Concise and practical** - optimize for recall during an interview, not textbook completeness.
3. **Always include examples** - code snippets, design scenarios, or answer outlines.
4. **Always include diagrams when useful** - use Mermaid or ASCII for system flows, algorithms, or story structure.
5. **Capture mistakes** - add brief practice notes when a test reveals a missed edge case, tradeoff, or weak answer.

## File Structure

1. **Topic with subtopics -> folder with README.md** - for larger clusters like arrays, caching, or leadership stories.
2. **Standalone topic -> single .md file** - when one file is enough.
3. **Every folder must have a README.md** - include scope, links, and how to practice.
4. **Use lowercase kebab-case filenames** - for example `two-sum.md` or `design-rate-limiter.md`.
5. **Separate source from answer** - notes should teach reusable patterns, not copy paid book text or proprietary prompt banks.
